微信本地数据自动化解析与 AI 处理工作站 (WeChat-Data-AI-Dashboard)
需求与技术实现文档 (PRD & Technical Design)
1. 项目背景与目标
本项目旨在解决日常高频使用微信带来的信息碎片化和精力消耗问题。通过逆向解析 windows或macOS 本地微信的 SQLite 加密数据库，提取核心聊天记录、群聊数据和联系人信息，并在本地搭建一个可视化的 Web Dashboard。结合开源/本地 AI大模型，实现信息的自动摘要、待办提取、群组分析和素材沉淀，打造一个高度自动化的私域运营与自媒体内容处理工作站。

2. 系统架构与技术栈
本项目采用“直接读取本地 SQLite 副本”的轻量化架构（无额外重量级数据库依赖），数据物理隔离在本地，确保绝对隐私。

目标运行环境: Windows 10/11 (x64 架构)和macOS (Apple Silicon M4 芯片)

解密与内存抓取层: Python 3 + lldb (或 C 语言扩展) + pysqlcipher3

Web 后端服务: Node.js + better-sqlite3 + Express/Fastify (提供 RESTful API)

Web 前端展示: React / Vue 3 + TailwindCSS (轻量级 Dashboard UI)

AI 自动化集成: 预留接口，与本地或云端大模型（如 Claude、基于 OpenClaw 的定制 Agent）联动进行数据清洗与分析。

3. 核心技术实现细节
3.1 内存扫描与解密密钥提取 (macOS M4 环境)
这是项目的技术难点。因 macOS 的 SIP 机制，需编写 Python/Shell 脚本实现以下自动化流程：

脱壳重签名: 对 /Applications/WeChat.app 执行 Ad-hoc 重签名，剥离强安全限制：sudo codesign --force --deep --sign - /Applications/WeChat.app

进程注入与内存 Dump: 在微信运行态下，使用 lldb 附加到微信进程，或编写基于 <mach/mach.h> 的 C 扩展，扫描进程内存寻找匹配 64位 Hex 密钥 + 32位 Salt 的特征码。

密钥派生缓存: 提取出 Raw Key 后，由 Python 脚本将其安全保存在本地 .env 文件中供后续连接使用。

3.2 数据库连接与查询 (Node.js 层)
后端使用 better-sqlite3 连接解密后的 .db 副本。必须在连接初始化时执行以下 SQLCipher 4 兼容的 PRAGMA 指令：

SQL
PRAGMA key = "x'提取到的RAW_KEY'";
PRAGMA cipher_page_size = 4096;
PRAGMA kdf_iter = 256000;
PRAGMA cipher_hmac_algorithm = HMAC_SHA512;
PRAGMA cipher_kdf_algorithm = PBKDF2_HMAC_SHA512;
注：为了防止文件锁定冲突，Node.js 后端应当对原始 .db 和 -wal 文件进行热拷贝，对副本进行只读查询。

4. 核心功能模块与数据流向
模块一：垂直领域信息雷达与素材库 (Content Radar)
数据源: MSG.db 中的 Type 49（公众号推文/外部链接/小程序分享）和 Type 1（群聊文本）。

需求逻辑:

监听特定高价值群聊，自动提取关于 AI、编程等关键词的长文讨论和链接。

利用 AI 接口将提取的链接内容生成摘要（Summary）。

在 Dashboard 上形成结构化的“灵感素材库”，支持一键导出为微信公众号推文或短视频脚本的初稿。

模块二：私域流量与群活跃度看板 (Group Analytics)
数据源: MSG.db 结合 MicroMsg.db (联系人/群组映射)。

需求逻辑:

统计特定粉丝群的每日发言量、活跃用户 TOP 10。

利用 AI 对群聊内容进行词频分析，提取群友近期的“技术痛点”、“报错求助”关键词。

在前端生成图表，辅助发现全网首发教程的痛点选题。

模块三：核心待办与文件管理中心 (Todo & File Center)
数据源: MSG.db 中包含 @当前用户 标记的消息，以及 Type 49 中的文件传输记录。

需求逻辑:

自动抓取并过滤出所有被 @ 的消息以及接收到的重要文件（如代码包、文档）。

提取文件在 Mac 本地的绝对路径，在网页上提供一键“在 Finder 中揭示”或直接预览功能。

将未处理的 @ 消息和文件统一转换为可勾选的 Todo List。

模块四：消息防撤回与全量复盘 (Anti-Revoke)
数据源: MSG.db 中的 Type 10000 (系统消息) 及热备份的数据库文件。

需求逻辑:

Node.js 后端定时（或基于文件系统事件监听）同步 .db-wal 文件的增量数据。

当检测到 Type 10000 的撤回操作时，拦截该操作在前端的影响，通过 CreateTime 和 MsgSvrID 关联回原始消息记录，并在 Dashboard 界面上以特殊样式（如红色删除线并保留原文）展示已撤回内容。

5. 数据库表结构参考指引 (AI 开发参考)
开发时请重点分析以下数据表：

MSG.db -> message 表: 关注 CreateTime (时间戳), Type (消息类型), Des (接收者/发送者关联), Content (XML/JSON/纯文本内容)。

MicroMsg.db -> Contact 表: 关注 UserName (wxid), NickName (昵称), Remark (备注)。

Session.db -> Session 表: 关注未读消息数和当前活跃的会话排序。
